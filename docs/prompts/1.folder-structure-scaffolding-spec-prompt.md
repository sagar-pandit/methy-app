# Specification Prompt: Folder Structure Scaffolding

## Feature Description

Scaffold the complete Methy food delivery platform folder structure including all microservice projects, Angular frontend, shared libraries, infrastructure configuration, CI/CD workflows, documentation structure, and observability setup — ready for Phase 1 development to begin.

## Context

- **Phase**: 1 (Foundation, Weeks 1-4)
- **Constitution**: v1.0.0 (Learning First, Simplicity, Test-Driven, API-First, Security, Observable, Local-First)
- **Source**: Product Vision document (Approved, v1.0)

## Requirements

### Backend (.NET 10 LTS)

Create a .NET solution with the following microservice projects:

1. **Methy.ApiGateway** — API Gateway (routing, authentication, rate limiting)
2. **Methy.Services.User** — User Service (auth, registration, profiles)
3. **Methy.Services.Restaurant** — Restaurant Service (CRUD, menu management)
4. **Methy.Services.Order** — Order Service (placement, tracking, history)
5. **Methy.Services.Payment** — Payment Service (mock/test mode processing)
6. **Methy.Services.Notification** — Notification Service (email, push, in-app)
7. **Methy.Services.Delivery** — Delivery Service (assignment, tracking, status)
8. **Methy.Shared** — Shared library (DTOs, contracts, common utilities, events)

Each service must include:
- ASP.NET Core Web API project
- Corresponding `.Tests` XUnit test project
- Dockerfile (multi-stage build)
- `appsettings.json` and `appsettings.Development.json`
- Health check endpoint placeholder

### Frontend (Angular 19 + Material Design 3)

Create Angular workspace with:
- Angular Material configured with custom theme
- Core module (auth, guards, interceptors)
- Shared module (common components, pipes, directives)
- Feature modules placeholder folders (restaurants, orders, delivery, admin)
- Environment configuration (local development)
- PWA manifest placeholder
- Karma/Jest test configuration

### Infrastructure (Local-First, $0 cost)

- `docker-compose.yml` — Full local environment (all services + dependencies)
- `docker-compose.override.yml` — Development overrides
- Minikube Kubernetes manifests folder (`k8s/`)
  - Namespace, deployments, services, ingress, configmaps
- RabbitMQ configuration
- SQL Server initialization scripts folder
- Nginx/Traefik ingress configuration

### CI/CD (GitHub Actions)

- `.github/workflows/build.yml` — Build and test all services
- `.github/workflows/docker.yml` — Build and push Docker images
- `.github/workflows/deploy.yml` — Deploy to local/staging placeholder
- Branch protection rules documentation

### Observability

- Grafana dashboard provisioning folder
- Prometheus configuration (`prometheus.yml`)
- Jaeger configuration
- Logging configuration (Serilog)

### Documentation

- `docs/` folder structure:
  - `architecture/` — ADRs, diagrams, design decisions
  - `api/` — OpenAPI/Swagger contract stubs per service
  - `guides/` — Setup guides, contribution guidelines
  - `prompts/` — Existing prompts (keep current files)

### Root Configuration Files

- `.gitignore` (comprehensive for .NET, Angular, Docker, IDE files)
- `.editorconfig` (consistent coding style)
- `README.md` (project overview, quick start, prerequisites)
- `global.json` (.NET SDK version pinning)
- `.env.example` (environment variables template, no secrets)
- `Directory.Build.props` (shared MSBuild properties)
- `Directory.Packages.props` (centralized NuGet package management)

## Constraints (from Constitution)

- **Principle I**: Structure must be self-documenting and educational
- **Principle II**: No unnecessary abstractions — only folders that will be used in Phase 1-2
- **Principle III**: Test projects alongside source projects
- **Principle IV**: API contracts folder ready for OpenAPI specs
- **Principle V**: No secrets in any committed files; `.env.example` only
- **Principle VI**: Observability config ready from day one
- **Principle VII**: Zero cloud dependencies; all infrastructure runs locally

## Acceptance Criteria

1. Running `dotnet build` from solution root compiles all projects without errors
2. Running `ng serve` from frontend folder starts Angular dev server
3. Running `docker-compose up` brings up infrastructure dependencies (SQL Server, RabbitMQ)
4. Running `dotnet test` discovers and runs (empty) test projects
5. GitHub Actions workflow files pass YAML validation
6. `.gitignore` correctly excludes build artifacts, secrets, IDE files
7. All Dockerfiles build successfully (even if services return 503 initially)
8. Folder structure is navigable and purpose of each directory is clear from naming

## Out of Scope

- Actual business logic implementation (that's Phase 2+)
- Database migrations or seed data (just empty script folders)
- Fully configured Kubernetes deployments (just manifest placeholders)
- Angular component implementation (just module/folder structure)
- CI/CD pipeline logic beyond skeleton structure
